# Repository Guidelines

## Project Structure & Module Organization
This repository is a ROS 2 workspace rooted at `ros2_ws/`.
- `src/my_robot_controller/`: Python ROS 2 nodes (e.g., `my_first_node.py`, `turtle_controller.py`) and linter tests under `test/`.
- `src/so_arm_control/`: Python control package with launch/config files (`launch/bringup.launch.py`, `config/controllers.yaml`) and linter tests.
- `src/so_arm100_description/`: CMake-based robot description package (`urdf/`, `meshes/`, `launch/`, `rviz/`).
- `build/`, `install/`, `log/`: generated by `colcon`; do not hand-edit or commit build artifacts.

## Build, Test, and Development Commands
Run from workspace root:
- `colcon build --symlink-install`: build all packages for development.
- `source install/setup.bash`: overlay the workspace in your shell.
- `colcon test && colcon test-result --verbose`: run test/lint checks and print detailed results.
- `ros2 launch so_arm100_description display.launch.py`: visualize the arm model in RViz.
- `ros2 launch so_arm_control bringup.launch.py`: start ros2_control bringup.
- `ros2 run my_robot_controller test_node`: run a Python node entry point.

## Coding Style & Naming Conventions
- Python: follow PEP 8 + PEP 257; linting is enforced with `ament_flake8` and `ament_pep257`.
- Use 4-space indentation, `snake_case` for modules/functions, and clear ROS node/executable names in `setup.py` entry points.
- Keep launch files named `*.launch.py`; keep YAML controller configs in `config/`.
- CMake package (`so_arm100_description`) compiles with strict warnings (`-Wall -Wextra -Wpedantic`).

## Testing Guidelines
- Existing tests are primarily linter gates in each Python package under `test/test_*.py`.
- Add tests adjacent to package test folders with `test_*.py` naming.
- Before opening a PR, run `colcon test` and ensure `colcon test-result --verbose` shows no failures.

## Commit & Pull Request Guidelines
- Follow existing history style: short, imperative, sentence-case summaries (example: `Set up so_arm ros2_control bringup and controller configs`).
- Keep commits focused by package/feature.
- PRs should include:
  - clear description of behavior changes,
  - affected package(s) and launch/run commands used for validation,
  - linked issue (if applicable),
  - screenshots/RViz captures for visualization or bringup changes.
  
## Teaching Preference

The user is learning ROS 2.

Preferred AI behavior:

1. Explain ROS concepts before writing code.
2. Provide architecture explanations.
3. Break down problems into steps.
4. Encourage the user to implement code themselves.

Avoid:

- writing entire packages automatically
- overly complex abstractions

## Learning Mode (Important)

This repository is primarily a **learning project for ROS 2**.

The goal is for the user to understand:

- ROS 2 workspace structure
- nodes
- launch files
- ros2_control
- URDF robot description
- RViz visualization
- controller configuration

AI agents should **not automatically implement full solutions**.

Preferred behavior:

- explain concepts first
- guide the user step-by-step
- suggest partial code or pseudocode
- allow the user to write most of the implementation

Avoid generating large fully-completed modules unless the user explicitly requests it.

## Future Project Direction

This project may expand to include:

- simulation tools

AI agents should keep the code structure flexible for these additions.
